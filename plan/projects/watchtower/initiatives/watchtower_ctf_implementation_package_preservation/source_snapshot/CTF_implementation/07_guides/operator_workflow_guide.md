# Operator Workflow Guide

## Intended Flow

1. intake the challenge and normalize the challenge root;
2. confirm environment and mode/autonomy assumptions;
3. run the active challenge loop with command capture, evidence capture, and notes updates;
4. enter blocker recovery when progress stalls materially;
5. close the challenge with solution, recap, closeout record, and extraction output;
6. review candidate reusable knowledge before promotion.

## Operator Expectations

- `notes.md` is the live working surface;
- `challenge.md` remains source-faithful;
- `challenge.md` front matter may appear after intake at the next workflow opportunity rather than immediately;
- environment and mode changes must be explicit;
- user-created ad hoc notes sections are allowed during active work and normalized only before closeout;
- evidence and provenance matter even in CTF workflows when the pack records claims, solutions, and reusable guidance.

## Mode Expectations

- assistant: guided reasoning and recommendations;
- note-taker: passive structured recording;
- teacher: explanatory operating mode focused on explanation rather than direct answers;
- full-auto: strongest execution posture and strongest confirmation/audit burden, with planned-command previews and concise post-step summaries.
