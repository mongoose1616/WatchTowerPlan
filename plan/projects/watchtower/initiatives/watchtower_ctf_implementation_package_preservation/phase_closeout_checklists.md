# WatchTower CTF Phase Closeout Checklists

## Summary

These checklists are the short final pass for each preserved phase. They do not replace `implementation_slice.md`, `phase_output_manifest.md`, or the live task records. Use them right before marking a phase complete.

## Global Rules

- Check the phase task record under `.wt/tasks/**/task.json` first.
- Keep docs, machine artifacts, indexes, rendered surfaces, and evidence aligned in the same change set.
- If a checklist item fails because the preserved baseline is wrong, record a new live-contract delta before changing the baseline.

## `phase.0` Shared Contract Adoption And Alignment

- The current-compatible identity, donor or recipient split, and source-of-truth order are still consistent across the canonical docs.
- `README.md`, `cold_start_runbook.md`, `phase_output_manifest.md`, `conditional_revisit_queue.md`, and `contradiction_sweep_ledger.md` agree on the current handoff posture.
- No hidden blocker remains before `/home/j/WatchTower` mutation begins.
- Coordination still points at the same real next engineer action.

## `phase.1` Recipient Scaffold And Bootstrap

- The recipient repo contains the staged shared core and the first scaffolded `offensive_security/` root.
- Required starter `README.md` and `AGENTS.md` surfaces exist at the mandated roots.
- Starter workflow metadata is gone before route preview, workflow indexing, or workflow discovery are treated as valid.
- Scaffold, bootstrap, pack validate, and validate-all proof are captured.

## `phase.2` Pack Machine Contract

- The required schemas, registries, policies, and baseline validation suite exist and validate.
- `template_catalog`, `documentation_family_registry`, and `human_surface_policy_registry` are real artifacts, not only design prose.
- The authored registries still align with `starter_registry_exemplars.md` or document the intentional divergence.
- The first governed record shapes and first machine surfaces still align with `artifact_specimens.md` and `machine_surface_specimen_index.md`.
- No Phase 3 runtime work still depends on re-deriving field meaning from donor prose.

## `phase.3` Runtime, Query, Sync, And Workflow Seam

- The first vertical slice passes the exact proof boundary in `vertical_slice_proof_spec.md`.
- Workflow docs, workflow metadata, routing, command docs, queries, and sync targets agree.
- Query and sync claims are backed by real pack outputs rather than placeholders.
- The Phase 3 validator and smoke-test expectations in `phase_test_matrix.md` are satisfied explicitly.
- The first unit tests and CLI smoke tests exist immediately after the slice proof, not as a later undefined follow-up.

## `phase.4` Challenge Artifacts And Closeout

- One challenge lifecycle proves `challenge.md`, `notes.md`, evidence capture, discrepancy handling, and closeout together.
- Closeout cannot pass while a discrepancy carries a no-closeout rule.
- `.wt_local/` is frozen in place at closeout rather than relocated or deleted.
- Artifact and closeout records validate and match the preserved field-level contracts.

## `phase.5` Knowledge Promotion And Retrieval

- Challenge-specific detail is stripped or explicitly quarantined before promotion.
- Typed relations remain authoritative on source artifacts, not only in derived indexes.
- Promotion and `review_status` stay distinct in both docs and implemented artifacts.
- The implemented candidate-to-accepted flow still matches `promotion_extraction_map.md`.
- Retrieval ranking and match explanations are proven against real promoted or candidate knowledge.

## `phase.6` Environment Adapters And Safety

- Every environment mode has explicit allowed actions, refusals, confirmation gates, and provenance behavior.
- Requested versus effective mode is visible in session state and event stream where required.
- Full-auto observability is proven before unattended execution claims are accepted.
- No high-risk path bypasses actor-ref, confirmation, or transfer-governance requirements.

## `phase.7` Release And Portability Proof

- Export and portability proof use staged export artifacts rather than a raw donor worktree.
- The supported handoff modes are explicit: `core-only`, `core-plus-pack`, and `pack-only`.
- Bootstrap, pack validate, validate all, and release checks are reproducible from documented commands.
- Release guidance no longer depends on unstated donor-repo cleanup or local workspace assumptions.
