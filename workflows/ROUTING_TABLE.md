# Routing Table

Use this table to select the minimum workflow modules required for a request. Treat the trigger keywords as examples, not as the only routing mechanism: route from the full prompt context, infer the nearest task type or task types, then load `modules/core.md` plus only the additional modules needed for that result.

Workflow modules are repository-available building blocks, but they are not active just because they exist under `workflows/modules/`. A module becomes active only when the routing result selects it, the user explicitly asks for it, or an active route merges it because new scope or risk is discovered during execution.

If multiple task types match one request, load the minimum union of their module sets and deduplicate shared phases instead of inventing a broad catch-all route.

When a request includes explicit commit intent such as `git commit`, `create commit`, `group commit`, `atomic commit`, `commit pass`, or `close out changes`, merge `modules/commit_closeout.md` into the dominant route instead of treating commit creation as an unrelated follow-up. If the request is only to commit an already-complete change set, use the Commit Closeout route by itself. Do not load `modules/commit_closeout.md` merely because a task might be committed later.

During a non-documentation workflow, if a documentation gap is discovered, keep the current module set and merge the smallest documentation route needed. Add the relevant documentation leaf module (`modules/documentation_generation.md` for new docs or `modules/documentation_refresh.md` for existing docs) plus any required shared phase modules from that route that are not already loaded. If the gap is not handled in the same change, record it as explicit follow-up work.

When a task's main risk is drift between implementation behavior and companion docs or machine-readable lookup surfaces, use the Documentation-Implementation Reconciliation route or merge `modules/documentation_implementation_reconciliation.md` into the active route rather than relying only on informal handoff checks.

When a task's main risk is drift between traced planning or governance artifacts and their companion trackers, family indexes, or unified traceability joins, use the Traceability Reconciliation route or merge `modules/traceability_reconciliation.md` into the active route rather than relying only on planning leaf workflows or handoff review.

When a task's main risk is drift between schema-backed governed artifacts and their companion schemas, examples, indexes, registries, or loader and validator assumptions, use the Governed Artifact Reconciliation route or merge `modules/governed_artifact_reconciliation.md` into the active route rather than relying only on generic validation.

When a request both distills external source material and asks for a new repository document, merge the Reference Distillation and Documentation Generation routes rather than treating either route as sufficient by itself.

When a request creates, updates, splits, blocks, unblocks, or closes local task records, use the Task Lifecycle Management route or merge `modules/task_lifecycle_management.md` into the active route rather than treating task-state updates as informal notes.

When a request explicitly hands work from one owner or phase to another, use the Task Phase Transition route or merge `modules/task_phase_transition.md` into the active route rather than relying only on generic handoff review.

When a request explicitly asks to align documentation or workflow guidance with repository foundations, use the Foundations Alignment Review route or merge `modules/foundations_context_review.md` with `modules/documentation_refresh.md` rather than treating foundations alignment as an implicit side note.

When a request needs hosted GitHub issue or project visibility for local task records, use the GitHub Task Sync route or merge `modules/github_task_sync.md` into the active route rather than treating the sync command as an ad hoc manual follow-up.

## Reconciliation Quick Guide

| Primary Drift Boundary | Preferred Route | Typical Surfaces |
|---|---|---|
| Implementation behavior versus companion docs or machine-readable lookup surfaces | `Documentation-Implementation Reconciliation` | code, tests, CLI help, command pages, README files, examples, lookup indexes |
| Traced planning or governance artifacts versus trackers, initiative views, family indexes, or trace joins | `Traceability Reconciliation` | PRDs, designs, plans, decisions, task or initiative trackers, traceability and initiative indexes |
| Schema-backed artifact families versus schemas, examples, registries, indexes, loaders, or validators | `Governed Artifact Reconciliation` | contracts, schemas, registries, catalogs, examples, validators, loader assumptions |
| One trace's acceptance intent versus its acceptance contract, evidence, validator linkage, or trace coverage | `Acceptance and Evidence Reconciliation` | PRD acceptance IDs, acceptance contracts, validation evidence, validator registry, traceability coverage |
